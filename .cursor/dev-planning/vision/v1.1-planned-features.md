## Overview

This document tracks features explicitly deferred from v1.0.0 based on gap analysis between the v1.0.0 roadmap and the original protocol specification (`general-specs.md`).

All items below are **planned but not committed** to v1.1.0.

## Feature Details

### 1. WebSocket Transport Binding

**Source**: [general-specs.md §1, §6.3, §13.1](../../docs/general-specs.md)

**Current State**:
- v1.0.0 includes WebSocket **concept example** only (pseudocode in examples)
- Schema supports `events` endpoint for WebSocket (`wss://`)
- No actual WebSocket transport implementation

**Scope for v1.1**:
```
[ ] Define ASAP over WebSocket binding specification
[ ] Implement WebSocket transport in client
[ ] Implement WebSocket transport in server
[ ] Add connection management (reconnect, keepalive)
[ ] Streaming support via WebSocket
[ ] Integration tests
[ ] Performance benchmarks
```

**Related**: DD-001 in [prd-v1-roadmap.md](../prd/prd-v1-roadmap.md#dd-001-websocket-transport)

---

### 2. OAuth2/OIDC Integration

**Source**: [general-specs.md §10.2](../../docs/general-specs.md)

**Current State**:
- Schema supports `oauth2` in `AuthScheme`
- Documentation shows integration patterns
- Runtime validation **rejects** `oauth2` as unsupported scheme
- Bearer tokens work; OAuth2 flow does not

**Scope for v1.1**:
```
[ ] Implement OAuth2 client credentials flow
[ ] Implement OAuth2 authorization code flow
[ ] Token refresh handling
[ ] Integration with common providers (Auth0, Keycloak)
[ ] OIDC discovery support
[ ] Update middleware to accept oauth2 scheme
[ ] Examples with popular providers
```

**Decision Required**: Server vs client-side token handling

---

### 3. Webhook Callbacks

**Source**: [general-specs.md §6.3](../../docs/general-specs.md)

**Current State**:
- Spec defines `callback_url`, `callback_events`, `callback_secret`
- **Not implemented** in current codebase

**Scope for v1.1**:
```
[ ] Implement callback registration in TaskRequest
[ ] Implement callback delivery with HMAC signing
[ ] Retry logic for failed deliveries
[ ] Callback event filtering
[ ] Security: validate callback URLs
[ ] Rate limiting for callback endpoints
```

---

### 4. State Storage Interface

**Source**: [general-specs.md §13.2](../../docs/general-specs.md)

**Current State**:
- Spec lists this as **open decision**
- Currently implementation-specific (opaque)

**Scope for v1.1**:
```
[ ] Define abstract state storage interface
[ ] Reference implementation: in-memory
[ ] Reference implementation: SQLite
[ ] Reference implementation: Redis
[ ] Migration guide for custom implementations
```

**Options**:
1. Opaque (current) - Implementation chooses
2. Interface-defined - Abstract interface in spec
3. Reference implementations - SQLite/Redis examples

**Recommendation**: Option 2 for interoperability

---

### 5. Message Broker Integration (NATS/Pub-Sub)

**Source**: [general-specs.md §6.3, §7.4](../../docs/general-specs.md)

**Current State**:
- Explicitly excluded from v1.0.0 PRD
- Spec defines `broker` config with NATS example
- "Mesh" topology pattern requires broker

**Scope for v1.1**:
```
[ ] NATS transport binding
[ ] Subject-based routing
[ ] Reply subject handling
[ ] Connection pooling for broker
[ ] Dead letter queue handling
[ ] (Optional) Kafka support
```

**Note**: May be better as external package (`asap-nats`)

---

### 6. DNS-SD Discovery

**Source**: [general-specs.md §7.4](../../docs/general-specs.md)

**Current State**:
- Spec mentions `_asap._tcp.example.com`
- Listed as "deferred" in MVP scope
- Well-known URI discovery is primary method

**Scope for v1.1**:
```
[ ] DNS-SD service type registration
[ ] mDNS/Bonjour support for local discovery
[ ] Integration with service mesh (Consul, etc.)
```

**Priority**: Lower - well-known URI covers most use cases
